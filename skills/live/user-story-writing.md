## Summary
Write user stories that focus on the user, not the solution, so the team stays aligned on why we are building something.

## When to use this skill
- You are defining work for a sprint or iteration.
- You need to hand requirements to design or engineering.
- You want to prevent scope creep during development.

## The playbook
1. **Identify the persona** — who is the user? Be specific.
2. **State the action** — what are they trying to do?
3. **Declare the outcome** — what changes for them when this works?
4. **Add acceptance criteria** — 3–5 concrete, testable conditions.
5. **Share for review** — before it goes to the team, sanity-check with a peer.

## Template

> As a **[persona]**, I want **[action]**, so that **[outcome]**.

## Common anti-patterns
- Writing stories about the system instead of the user.
- Skipping acceptance criteria because "everyone knows what I mean."
- Making stories too big. If it takes more than a few days, split it.

## Exit criteria
- A developer can read the story and ask clarifying questions, not fundamental ones.
- You can delete the story and still know what problem you were solving.
